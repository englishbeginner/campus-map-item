# 校园移动端导航系统开发文档
> 文档版本：V1.0  
> 编写日期：2026年3月  
> 项目状态：初始开发阶段

## 1. 项目概述
### 1.1 项目背景
当前通用地图软件（如高德、百度）在大学校园场景下存在数据精度不足、建筑标识模糊、校内道路更新不及时、校车信息缺失等痛点。本项目旨在开发一款专属的校园移动端导航系统，解决师生“找路难、等车难”的问题，并为后续校园生活服务生态预留扩展接口。

### 1.2 项目目标
- 短期目标：实现校园内建筑的精确定位与可视化、校园路径规划、简化版校车信息查询。
- 长期目标：打造集导航、生活服务、教务信息于一体的校园综合平台。

### 1.3 核心设计原则
- 移动端优先：所有交互与视觉设计以手机小屏幕体验为核心。
- 数据驱动：建立自有的POI（兴趣点）数据库，不完全依赖第三方地图底图数据。
- 轻量高性能：确保地图加载流畅，避免手机发热与卡顿。

## 2. 总体技术架构
本系统采用前后端分离的经典B/S架构。

### 2.1 架构分层
- 用户层：移动端H5网页（未来可扩展为微信小程序或App）。
- 接入层：前端通过HTTPS协议请求后端RESTful API。
- 业务逻辑层（后端）：基于Spring Boot构建，处理POI数据、导航计算、校车数据逻辑。
- 数据访问层：MyBatis-Plus负责与数据库交互。
- 数据存储层：MySQL存储结构化数据，Redis存储热点缓存。
- 第三方服务：高德/腾讯地图Web API（提供底图渲染、基础定位、路线规划引擎）。

### 2.2 技术选型清单
- 后端：Java 21, Spring Boot 3.2.5, MyBatis-Plus, MySQL 8.0, Redis 7.x。
- 前端：Vue 3.4, Vite 5, Vant 4, Pinia, 高德地图JS API v2.0。

## 3. 功能模块详细设计
### 3.1 核心地图与POI模块
#### 3.1.1 自定义底图渲染
- 功能描述：
    - 前端初始化地图SDK，加载第三方底图。
    - 通过地图服务商的自定义样式平台，配置校园专属样式：将校园外区域透明度降低至50%，校园内道路颜色提亮，放大校园内文字字号。
    - 坐标系统一使用GCJ-02（火星坐标），避免前端显示偏移。
- 开发要点：前端需监听地图的“缩放级别”和“视野变化”事件。

#### 3.1.2 校园POI展示与搜索
- 功能描述：
    - 视野内加载：当地图移动或缩放停止时，前端计算当前屏幕的经纬度范围（Bounds），请求后端接口。后端只返回该范围内的POI数据，以提升性能。
    - 分层渲染：
        - 低缩放级别：仅显示重点地标（如校门、图书馆）。
        - 高缩放级别：显示所有建筑轮廓和详细图标。
    - POI详情：点击地图上的标记，底部弹出详情卡片，展示该建筑的实景照片、名称、简介。
    - 搜索功能：用户在顶部搜索框输入关键词，后端进行模糊匹配（支持教学楼别名，如“一教”匹配“第一教学楼”），返回结果列表，点击后地图自动飞点（FlyTo）到该位置。

### 3.2 建筑精准可视化模块
#### 3.2.1 数据采集与录入
- 功能描述：
    - 不仅存储建筑的“中心点”，还需存储建筑的“轮廓坐标”（Polygon）。
    - 数据采集流程：
        1. 利用地图坐标拾取工具获取初步数据。
        2. 实地人工校准，确保大门、主要入口的坐标精准。
        3. 在后台管理系统中绘制或上传轮廓数组。

#### 3.2.2 前端视觉呈现逻辑
- 功能描述：
    - 绘制轮廓：前端拿到Polygon数据后，调用地图SDK的多边形绘制接口，为建筑填充半透明底色，描边。
    - 图标覆盖：在轮廓中心上方，覆盖分类SVG图标（教学楼用书本图标，食堂用碗筷图标）。
    - 交互优先级：点击事件优先响应图标，其次响应轮廓区域。

### 3.3 校园专属导航模块
#### 3.3.1 路径规划
- 功能描述：
    - 起点/终点设置：支持“我的位置”到“搜索结果”或地图选点。
    - 路线计算：
        - 前端将起终点坐标传给后端。
        - 后端优先请求第三方地图API获取步行路线。
        - 后端进行“路线吸附”逻辑判断：如果第三方路线偏离了数据库中存储的“校园内部道路”，则进行微调或返回预设的校园道路数据。
    - 多模式选择：默认步行，可选骑行。

#### 3.3.2 导航体验
- 功能描述：
    - 前端开启持续定位追踪。
    - 沿途语音/文字提示：“前方100米经过第二食堂”。
    - 到达提醒：距离目标50米时，触发手机震动提示。

### 3.4 校车运营服务模块
#### 3.4.1 线路与站点管理
- 功能描述：
    - 数据库存储线路基本信息（名称、首末班时间）。
    - 存储站点的有序列表（关联线路ID，包含序号、站点名称、坐标）。
    - 存储线路的轨迹坐标串（用于在地图上画线）。

#### 3.4.2 前端展示逻辑
- 功能描述：
    - 地图下方采用Tab栏切换不同线路（如A线、B线）。
    - 选中某线路后，地图清除其他图层，仅绘制该线路的轨迹线和站点标记。
    - 点击站点，显示该站的预计发车时间（基于时刻表的简单计算）。

### 3.5 后台管理模块
- 功能描述：
    - POI管理：增删改查POI信息，支持上传实景图，支持在地图可视化界面上拖拽调整坐标。
    - 校车管理：编辑线路信息，上传轨迹文件，维护发车时刻表。
    - 数据统计：简单的PV/UV统计。

## 4. 数据库设计
### 4.1 核心数据表结构
#### 4.1.1 校园POI表 (t_campus_poi)
| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| id | BIGINT | PK, Auto Increment | 主键 |
| name | VARCHAR(100) | NOT NULL | 建筑/场所名称 |
| alias | VARCHAR(100) | NULL | 别名（用于搜索，如“一教”） |
| type | TINYINT | NOT NULL, INDEX | 类型：1-教学楼，2-食堂，3-宿舍，4-图书馆，5-校门，6-其他 |
| center_lng | DECIMAL(10,7) | NOT NULL | 中心点经度 |
| center_lat | DECIMAL(10,7) | NOT NULL | 中心点纬度 |
| boundary | TEXT | NULL | 建筑轮廓坐标数组（JSON格式存储） |
| photo_url | VARCHAR(255) | NULL | 实景照片URL |
| description | VARCHAR(500) | NULL | 简介 |
| status | TINYINT | DEFAULT 1 | 状态：0-禁用，1-启用 |
| create_time | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | DEFAULT CURRENT_TIMESTAMP ON UPDATE | 更新时间 |

#### 4.1.2 校车线路表 (t_bus_route)
| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| id | BIGINT | PK | 主键 |
| route_name | VARCHAR(100) | NOT NULL | 线路名称（如：A线 北门-南门） |
| start_time | TIME | NULL | 首班车时间 |
| end_time | TIME | NULL | 末班车时间 |
| path_coords | TEXT | NULL | 行驶轨迹坐标串 |
| status | TINYINT | DEFAULT 1 | 状态 |

#### 4.1.3 校车站点表 (t_bus_stop)
| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| id | BIGINT | PK | 主键 |
| route_id | BIGINT | NOT NULL, INDEX | 所属线路ID |
| stop_name | VARCHAR(100) | NOT NULL | 站点名称 |
| lng | DECIMAL(10,7) | NOT NULL | 经度 |
| lat | DECIMAL(10,7) | NOT NULL | 纬度 |
| sort_order | INT | NOT NULL | 站点排序序号 |

#### 4.1.4 系统用户表 (t_sys_user)
| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| id | BIGINT | PK | 主键 |
| username | VARCHAR(50) | NOT NULL, UNIQUE | 管理员账号 |
| password | VARCHAR(100) | NOT NULL | 加密后的密码 |
| role | VARCHAR(20) | DEFAULT 'ADMIN' | 角色 |

### 4.2 数据库索引设计
- t_campus_poi 表：在 type 和 status 上建立联合索引，用于快速筛选某类建筑。
- t_bus_stop 表：在 route_id 上建立索引，确保查询某条线路的站点时速度极快。

## 5. 系统接口设计
### 5.1 接口规范
- 协议：HTTPS
- 数据格式：JSON
- 鉴权方式：后台管理接口使用JWT Token鉴权；地图公开数据接口无需鉴权（或使用API Key限流）。

### 5.2 核心接口定义
- POI接口：
    - GET /api/v1/pois/list-in-bounds：根据经纬度范围获取POI列表。
    - GET /api/v1/pois/search：关键词搜索POI。
    - GET /api/v1/pois/{id}：获取POI详情。
- 校车接口：
    - GET /api/v1/bus/routes：获取所有线路列表。
    - GET /api/v1/bus/routes/{id}/stops：获取某线路的所有站点及轨迹。
- 后台管理接口：
    - POST /api/admin/poi：新增POI。
    - PUT /api/admin/poi/{id}：修改POI。

## 6. 开发流程与部署
### 6.1 开发阶段划分
1. 第一阶段：环境搭建与基础功能
- 后端：搭建Spring Boot工程，集成MyBatis-Plus，设计并创建数据库表，实现POI的CRUD接口。
- 前端：搭建Vue3+Vite工程，引入Vant和地图SDK，实现地图加载、定位、展示后端返回的Marker。
2. 第二阶段：核心功能实现
- 后端：实现POI范围查询算法、对接第三方地图WebClient、实现校车数据接口。
- 前端：实现搜索功能、绘制建筑轮廓、实现校车线路Tab切换、集成路径规划。
3. 第三阶段：后台管理与优化
- 开发简易的后台管理页面（建议使用Vue3+Element Plus）。
- 性能优化：加入Redis缓存热点POI数据，前端实现懒加载。
- 实地测试与数据校准。

### 6.2 部署方案
- 服务器：推荐使用Linux服务器（CentOS/Ubuntu）。
- 部署方式：
    - 后端：使用Docker容器化部署，或直接打包Jar包运行。
    - 前端：打包后生成的dist文件夹，通过Nginx进行静态资源代理。
    - 数据库：MySQL主从（可选），Redis单机或哨兵模式。
